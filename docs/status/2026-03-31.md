# Status as of 2026-03-31

## Installation validation in VM

- The first test `QEMU + Arch ISO + Margine` found two real bugs:
live repo mount under `/mnt` and no `/etc/vconsole.conf` during
the generation of `mkinitcpio` images.
- The flow has been corrected: the live repo must be mounted outside the root target,
bootstrap installs `vconsole.conf` before the boot chain and chroot
initial skips `snap-pac` post-hooks.
- The VM completed a successfully recovered installation, producing:
  `BOOTX64.EFI`, `limine.conf`, `margine-linux.efi`,
  `margine-linux-fallback.efi`, and `margine-recovery.efi`.

## Display scaling and refresh rate

- On the real system and in `Margine`, a stricter display choice has been
  confirmed: no default fractional scaling on external outputs.
- `DP-11` remains at `2.0` and `xwayland.force_zero_scaling = true` is treated
  as a standard mitigation for mixed Wayland/XWayland rendering defects.
- `misc:vrr = 2` has been enabled as a best-effort option for monitors that
  really support `Adaptive Sync`, but it is not confused with power saving.
- For the Framework 13 laptop an explicit solution has been added for the
internal panel: a user service that switches automatically
`eDP-1` to `2880x1920@60` on battery and to `2880x1920@120` when the machine is
powered.

## Implications for Margine

- Target behavior no longer depends on manual commands after login.
- The user runtime layer now also installs and enables
  `hypr-refresh-rate.service`, alongside `elephant.service`.
- The final desktop strategy is therefore:
  full scaling, `XWayland` mitigation, `VRR` only as optional monitor support,
  and an explicit refresh-rate switch for the laptop's internal panel.

## Power baseline Framework

- On the real system `power-profiles-daemon` exposes explicit actions for the
AMD panel (`amdgpu_panel_power`) and for GPU power management
  (`amdgpu_dpm`).
- The policy found to make sense is now made explicit in `Margine`:
  `battery_aware=true`, base profile `balanced`,
  `amdgpu_panel_power=false`, `amdgpu_dpm=false`.
- `Margine` does not introduce a watcher that continuously overwrites the profile
CPU: the power profile remains the responsibility of `power-profiles-daemon`, while the
panel refresh rate is managed separately.

## Hyprland Session, Walker, and Elephant

- On the real system the diagnosis was confirmed: the problem is not
`Walker` by itself, but the fact that `Elephant` can launch GUI through
  `systemd-run --user --scope`.
- The correct distro-level fix for `Margine` is therefore the explicit import
  of the Hyprland graphical environment into the `systemd --user` manager and
  D-Bus activation, very early in session startup.
- The solution has been versioned as a separate script, called by
`hyprland.conf`, instead of remaining as a local workaround or app-specific fix.
