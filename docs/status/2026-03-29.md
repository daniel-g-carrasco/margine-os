# Stato al 2026-03-29

## Decisioni prese oggi

- nome progetto umano: `Margine`
- nome repository: `margine-os`
- approccio repo-first, non iso-first
- documentazione obbligatoria
- approccio didattico obbligatorio
- replica del sistema tramite allowlist, non tramite dump completo
- `Floorp` escluso
- `Firefox` incluso come browser principale
- AUR consentito solo come eccezione motivata
- `Koofr` è candidato AUR da valutare esplicitamente
- ADR 0002 accettato:
  - `Limine`
  - `UKI`
  - `LUKS2`
  - `TPM2`
  - `Btrfs`
  - `Snapper`
  - `Secure Boot` come obiettivo esplicito
  - snapshot bootabili come obiettivo esplicito
  - fallback a `systemd-boot` se la validazione fallisce
- ADR 0003 accettato:
  - `ESP` grande montata su `/boot`
  - resto del disco in `LUKS2`
  - root `Btrfs`
  - subvolumi dedicati per recovery, VM e container
  - `zram` come swap della `v1`
  - niente ibernazione nella `v1`
  - `compress=zstd:3` come base
  - `fstrim.timer` al posto di `discard`
  - distinzione esplicita tra workload `rootful` e `rootless`
  - manifest iniziale dei subvolumi target già fissato
- ADR 0004 accettato:
  - validazione per gate della catena `Limine + UKI + Secure Boot + TPM2 + Snapper`
  - `mkinitcpio` come punto di partenza per generare `UKI`
  - `sbctl` come gestore chiavi e firma
  - policy iniziale `TPM2` su `PCR 7+11`
  - recovery key obbligatoria prima dell'enrollment `TPM2`
- ADR 0005 accettato:
  - separazione tra percorso `prod` e percorso `recovery`
  - `UKI` normali con command line incorporata
  - `UKI` recovery senza command line incorporata
  - `Limine` in `EFI/BOOT`, `UKI` in `EFI/Linux`
  - `limine enroll-config` come parte della trust chain
- ADR 0006 accettato:
  - `Margine` come layer riproducibile sopra Arch rolling
  - nessun fork congelato di Arch
  - aggiornamento ordinario del sistema tramite normale flusso Arch
  - aggiornamento della repo solo quando cambia l'architettura o l'automazione
- ADR 0007 accettato:
  - `limine.conf` finale come artefatto generato
  - template versionato + facts macchina + blocco recovery
  - file finale sulla `ESP` non considerato sorgente autorevole
- ADR 0008 accettato:
  - `snapper + snap-pac` come baseline snapshot
  - snapshot automatici solo per `root`
  - timeline root disabilitata nella `v1`
  - `update-all` come percorso canonico sopra il safety net di `snap-pac`
  - consapevolezza esplicita che la `ESP` resta fuori dagli snapshot Btrfs
- ADR 0009 accettato:
  - `update-all` come orchestratore canonico
  - ordine esplicito delle fasi di update
  - distinzione tra errori hard e soft
  - primo script versionato di `update-all`
- ADR 0010 accettato:
  - deploy su `ESP` solo da artefatti gia' generati
  - backup preventivo dei file sovrascritti
  - nessuna pulizia distruttiva della `ESP` nella `v1`
- ADR 0011 accettato:
  - `limine enroll-config` eseguito dopo il deploy sulla `ESP`
  - firma `sbctl` eseguita dopo `enroll-config`
  - `BOOTX64.EFI` e `UKI` firmati come artefatti finali sulla `ESP`
  - `sbctl verify` come verifica finale del refresh trust
- ADR 0012 accettato:
  - bootstrap di Secure Boot separato da `update-all`
  - `sbctl enroll-keys -m` come default prudente
  - chiavi `sbctl` di tipo `file` nella `v1`
  - `Setup Mode` richiesto come prerequisito esplicito
