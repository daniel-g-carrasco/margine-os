# Because migrating does not mean cloning the home

When reinstalling a system, the natural temptation is:

- "I'll copy the whole home page and I'm done".

It seems convenient, but it's usually the best way to:

- drag bugs;
- keep useless files;
- lose understanding of what is really configuring the system.

The home page contains everything:

- useful configurations;
- cache;
- local databases;
- temporary state;
- files generated by specific apps;
- random identifiers.

In `Margine` we don't want to treat all of this as if it were equivalent.

## The correct criterion

The question is not:

- "does this file exist?"

The correct question is:

- "does this file deserve to become part of the target system?"

If the answer is yes, the file is:

- Understood;
- cleaned up;
- versioned.

If the answer is no, the file remains outside.

## Esempi

### Da migrare

- `~/.config/nvim`
- `~/.config/kitty/kitty.conf`
- `~/.config/user-dirs.dirs`

Why:

- they are legible;
- express true preferences;
- they make sense even on a new car.

### Do NOT migrate blindly

- entire browser profiles;
- editor cache;
- `userapp-*.desktop`;
- VS Code workspace storage;
- `.cache` directory.

Why:

- they bring in noise and an opaque state;
- they don't teach anything;
- they often worsen reproducibility.

## Rule of thumb

If you couldn't explain a file verbally in two minutes, it shouldn't go in
repo without review.
